## Context

CX Switch 现在的菜单栏面板已经具备账号切换、刷新、导入和设置入口，但信息组织更像功能清单，不像一个清晰的 dashboard。用户给出的参考图体现的是更强的分区层级：顶部当前账户摘要，中部可切换账户列表，底部动作区，这种结构更适合菜单栏应用的快速扫视和快速操作场景。

在这次改版里，界面分区之外还要补齐数据行为：每个账号的用量和摘要需要有独立缓存，面板打开时优先展示缓存结果；用户对单个账号触发刷新时，只更新目标账号，不把其他账户一起重新拉取，避免面板闪烁和多余请求。

## Goals / Non-Goals

**Goals:**
- 把菜单栏面板重构成清晰的三段式信息架构
- 让当前账户信息成为视觉锚点，优先暴露邮箱、计划类型、更新时间和用量
- 让账户切换列表保持高密度，但可读、可点、可扫视
- 让底部动作入口保持统一、稳定、低噪音
- 为每个账号引入独立缓存和独立刷新语义，减少重复请求和状态闪烁
- 保持账户数据来源、导入、切换和设置逻辑稳定，仅按新设计调整刷新行为

**Non-Goals:**
- 不引入新的存储格式或服务层协议
- 不改变 account-store、Keychain 或 codex app-server 的行为契约
- 不引入新的第三方依赖
- 不做大范围视觉风格重定向，只做分区和布局统一

## Decisions

### 1. 使用三段式固定结构，而不是单列堆叠式信息流

参考图最重要的特征不是颜色，而是层级分区：顶部概览、中部切换、底部动作。我们将面板拆成固定的三段，使用分隔线和垂直间距保证扫视路径明确。

Alternatives considered:
- 单列无分区布局：实现简单，但会让高频入口和信息混在一起，无法形成 dashboard 感
- 卡片堆叠布局：视觉更重，但会挤占菜单栏面板有限的宽度，降低可读性

### 2. 顶部当前账户区域作为“摘要卡”

顶部区域承担“我现在是谁、状态如何、最近是否更新”的职责，因此优先显示账户名/邮箱、计划类型、更新时间和用量。刷新动作保留在摘要区右侧，让用户能直接对当前账号做局部刷新操作。

Alternatives considered:
- 把刷新放到底部：会削弱当前账户区的自解释性
- 只显示邮箱不显示更新时间：信息密度更低，但失去参考图里的“Updated X seconds ago”提示

### 3. 中部账户列表采用紧凑行式布局

切换列表需要尽量节省垂直空间，同时保留可点击性。每行显示邮箱和计划类型，必要时补充一行用量条，保持与参考图一致的“轻量但完整”感觉。

Alternatives considered:
- 为每个账户使用完整卡片：会过于厚重，列表项太高
- 只显示邮箱文本：信息太少，无法在列表中快速判断账户状态

### 4. 底部动作区使用统一纵向动作列表

底部动作区是次级操作入口，不应该和账户切换列表混在一起。统一采用图标 + 文案的纵向列表，降低认知成本，也方便后续继续扩展动作项。

Alternatives considered:
- 水平按钮行：节省高度，但容易挤压菜单栏宽度，且层级不如纵向列表清晰
- 折叠式菜单：更省空间，但不符合参考图的直观暴露式布局

### 5. 每个账号使用独立缓存，并提供行内刷新入口

账户摘要和用量读取需要保持局部更新。我们将每个账号的最近一次快照视为独立缓存，面板打开时先渲染缓存，再按需刷新目标账号。这样做可以避免“刷新一个账号，整个面板一起抖动”的问题。每个账号行保留独立刷新入口，刷新语义只作用于该账号。

Alternatives considered:
- 全局统一刷新：实现简单，但会让所有账号一起重载，面板更容易闪烁
- 完全不缓存：可以减少状态复杂度，但在菜单栏场景里会显得慢且重复

## Risks / Trade-offs

- [窄宽度下邮箱和计划标签可能拥挤] → 使用截断、对齐和固定间距，必要时优先保留邮箱主信息
- [列表项信息增多导致面板变高] → 只保留当前账号和可操作账户的关键信息，用量条保持压缩呈现
- [界面重排影响既有交互节奏] → 保持按钮功能和状态逻辑不变，仅重组视觉结构
- [当前账号与保存列表信息重复] → 通过摘要区与切换区的职责区分减少重复感
- [账号缓存可能短暂过期] → 允许单账号手动刷新并在刷新完成后局部失效更新
- [单账号刷新入口可能增加视觉噪音] → 将刷新入口做成行内轻量图标，避免干扰主信息

## Migration Plan

1. 先在现有 SwiftUI 视图上重构分区，不改服务层和数据层
2. 在 `AppState` 中引入按账号缓存和单账号刷新能力，保持服务协议不变
3. 让当前账户摘要、账户切换列表、底部动作区逐步替换旧布局
4. 用现有数据源回放空状态、单账号、多账号、长邮箱和单账号刷新场景
5. 若布局在窄宽度下过密，优先收紧副信息，不回退到旧布局
