## Context

当前实现中，账号切换和 refresh token 导入都依赖 `account/read` 成功返回后才完成 UI 状态更新。这个链路把认证写入、服务重启、账号就绪、数据回填绑在一起，导致体感延迟高，而且在短时间连续切换时会放大卡顿和重复刷新。

## Goals / Non-Goals

**Goals:**
- 降低账号切换和导入后的主观等待时间
- 让 UI 先响应，再后台补齐真实账户状态
- 保持账号数据一致性，避免乐观更新覆盖掉认证和历史缓存
- 让单个账号刷新和其它账号交互互不阻塞

**Non-Goals:**
- 不改变 `codex app-server` 的协议
- 不移除现有 registry / auth.json 数据结构
- 不解决与切换无关的设置页或布局问题

## Decisions

### 1. 乐观切换优先于 live read

切换账号时，系统先写入目标账号的 auth，再立即把目标账号标记为当前账号并更新 UI。随后后台继续轮询 `account/read`，拿到 live 数据后再做合并。

**Alternatives considered:**  
继续阻塞直到 `account/read` 返回。这个方案最简单，但无法改善体感延迟，也容易让用户误以为点击无效。

### 2. 导入流程分为认证完成与数据回填两步

导入 refresh token 成功后，先确认 auth 文件已经写入并可用，再立即切换到新账号。账户详情、订阅档位和用量等信息留给后台同步补齐。

**Alternatives considered:**  
等完整账户信息返回后再展示导入结果。该方案更保守，但会继续保留长等待，且用户在导入成功后没有即时反馈。

### 3. registry 写入做短时合并

来自 live refresh、切换修复和导入回填的 registry 更新会在一个短窗口内合并，避免频繁写磁盘造成额外延迟。

**Alternatives considered:**  
每次状态变化都立即写盘。这样最直观，但会让高频切换和刷新时的 I/O 开销更明显。

### 4. 轮询采用退避而不是固定间隔

`account/read` 的 readiness 检查使用渐进式退避，减少无意义的重复请求，同时保留最终收敛的能力。

**Alternatives considered:**  
固定 1 秒轮询。实现简单，但在服务尚未准备好的阶段会放大等待感，也更容易造成重复请求。

### 5. 刷新状态按账号隔离

每个账号单独维护刷新状态，避免一个账号的加载态扩散到整个面板。这样 UI 仍然可操作，用户可以继续切换其它账号。

**Alternatives considered:**  
继续使用全局刷新态。该方式实现更快，但会让局部操作看起来像全局阻塞。

## Risks / Trade-offs

- [Risk] 乐观切换可能短暂显示缓存数据 → 通过后台 reconciliation 在 live 数据到达后尽快修正
- [Risk] 退避和合并写入会让状态落盘稍晚 → 通过短窗口策略和最终强制落盘降低一致性风险
- [Risk] 不同账号的数据在过渡期间可能短暂不完全一致 → 通过保留旧缓存和明确的加载态区分来降低困惑

## Migration Plan

1. 先实现乐观切换和导入两阶段流程，保持现有 auth 结构不变
2. 再引入 registry 合并写入和 account/read 退避
3. 最后把 UI 刷新状态改成按账号隔离，并验证快速连续切换
4. 若出现回归，回退到现有阻塞式流程即可，不需要数据迁移

## Open Questions

- readiness 轮询的最大退避窗口应设为多少秒
- 失败时是否需要在 UI 中显示一次性提示，还是仅保留状态栏文案
